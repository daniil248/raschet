# Архитектура «Конструктор схем» + дисциплины + специалисты

Статус: ЧЕРНОВИК (обсуждение, не реализация). Источник — интервью
2026-05-18 + мировые практики (Revit/BIM federated, EPLAN Platform,
SOLIDWORKS/Siemens single-source). Согласованные решения Пользователя
зафиксированы ниже; один открытый архитектурный форк помечен ❓.

## 1. Принцип

«Схема — всегда схема, независимо от типа.» Конструктор — это **общая
оболочка + движок-канва**, НЕ владелец всех данных и НЕ координатор.

Слои (логические, физическое расположение не меняется — закон границ
contracts/README.md в силе):

- **SHELL** — хром Конструктора: шапка (app-header), сайдбар-аккордеон,
  селектор страниц, зум (Ctrl+wheel), экспорт→reports, project-context,
  role-gate. Переиспользуется всеми дисциплинарными редакторами,
  включая СКС.
- **CANVA-ENGINE (node-graph)** — `js/engine/*`: узлы/связи/recalc/
  render/serialization. Используют электрика и scaffold-дисциплины
  (гидравлика/ОВиК/газ/пожаротушение) — палитры уже в
  `shared/discipline-context.js`.
- **SPECIALIZED-CANVA** — спец-канвы, НЕ node-graph: СКС «План зала»
  (метрический 2D, стойки/каналы/фитинги). Берёт SHELL, но сохраняет
  свой движок и механизмы.
- **DISCIPLINE-REGISTRY** — `shared/disciplines.js` (членство узла в
  дисциплинах, X.4.2) + `shared/discipline-context.js` (палитры/табы
  инспектора per-discipline). Уже существует — это и есть шов.
- **OBJECT-REGISTRY** — общий per-project реестр физических объектов
  (стойка/шкаф/устройство) с портами; дисциплины правят только свой
  срез атрибутов.
- **SPATIAL-BASE (архитектурная подложка)** — стены/площади/
  пространства/уровни/оси, которыми владеют роли **архитектор**
  (дисциплина `architectural`) и **конструктор** (structural ≈
  `mechanical`). Это базовый слой BIM-federated смысла: инженерные
  дисциплины НЕ создают геометрию здания, а ссылаются на неё
  (размещение стоек/оборудования, no-cross-стены cable-routing,
  площади помещений, коридоры). См. §10.

## 2. Типизация схемы (immutable)

При создании схемы Пользователь выбирает дисциплину/тип. **Тип
неизменяем после создания.** Существующий dropdown «Дисциплина схемы»
в свойствах проекта → read-only бейдж 🔒 после создания; выбор только
в мастере создания схемы/страницы.

**РЕШЕНО (форк закрыт интервью 2026-05-18): Вариант I —
страница = типизированная схема.** Проект — контейнер; каждая
страница несёт собственную immutable-дисциплину; один проект может
держать электро + ОВиК + газ страницы рядом. Ближе к memory D3
«универсальная база ВСЕХ типов».

Следствия для модели данных (объём миграции):
- `discipline` мигрирует scope project → page. Сейчас
  `state.project.discipline` (single, serialization default
  electrical). Цель: `page.discipline` (immutable, задаётся в
  мастере страницы).
- serialization: round-trip страниц с per-page discipline; backward-
  compat — старый project.discipline → дисциплина первой/единственной
  страницы (миграция с `typeof`-guard, memory:user-params-sacred).
- recalc/render/inspector — scope per active page-discipline (узлы
  страницы считаются её движком; cross-page — только через общий
  реестр объектов §6, не через граф).
- discipline-context.js (палитра/табы) переключается по
  active-page.discipline, а не project.discipline.

## 3. Специалисты: одна канва + фильтр дисциплины

(Q2 — выбрано.) Одна канва; селектор активной дисциплины (как уже в
шапке). Активная дисциплина — ярко и редактируемо; чужие слои —
приглушённая read-only подложка. Спец видит контекст, но не ломает
чужое. Палитра/табы инспектора переключаются по
`discipline-context.js`.

## 4. Роли (RBAC-расширение)

- Пользователю можно назначить НЕСКОЛЬКО ролей; эффективные права —
  объединение.
- Создание схемы гейтится дисциплинарной ролью: роль «электрик» →
  может создать ТОЛЬКО электрическую схему. Чужой тип — кнопка
  создания disabled + tooltip (не скрывать — паттерн RBAC из memory).
- Слой поверх существующего `shared/subscriptions.js` ROLES
  (manager/gip/engineer/viewer): добавить матрицу
  `disciplineCreate: { electrical, data, hvac, architectural, ... }`.
- **Технолог-ГИП — НЕ равный дисциплинарный специалист, а
  координатор** (memory D7). Полное управление проектом + функция
  ГИП на весь жизненный цикл объекта. Эффективные права = надмножество
  (видит все дисциплины read-write по координации, владеет lifecycle-
  статусом и агрегатами). Граница владения = акт ввода в эксплуатацию;
  далее — модуль эксплуатации/O&M (отдельная фаза ROADMAP).
- **Архитектор/конструктор — владельцы SPATIAL-BASE** (§10). Создают
  стены/пространства; инженерные роли получают их read-only как
  подложку (как чужие дисциплины в §3).

## 5. СКС: приведение к оболочке Конструктора

(Q1/Q3 — выбрано.) scs-design/scs-config:
- Принимают SHELL Конструктора (шапка/сайдбар-аккордеон/страницы/
  зум/экспорт/project-context/role-gate) — единый вид и эргономика.
- СОХРАНЯЮТ свой движок «План зала / Стойки / Связи» и все текущие
  механизмы (авто-раскладка, каналы, фитинги, длины кабеля).
- Остаются ЕДИНСТВЕННЫМ владельцем СКС-данных (шов project-storage,
  `data`-namespace). Конструктор показывает СКС только ПО ССЫЛКЕ
  (read-only снимок для координации), не редактирует.

⚠ Риск/решение: НЕ переносить «План зала» на node-graph движок
Конструктора (метрический 2D ≠ граф; высокий риск регрессий). Общая
только ОБОЛОЧКА, не движок канвы.

## 6. Общий реестр объектов + порт-driven видимость

(Q4 — выбрано; «текущая концепция остаётся».)
- Физический объект (стойка/шкаф/устройство) — ОДИН в per-project
  OBJECT-REGISTRY. Дисциплины правят только свой срез: электрик —
  `panelDesignation` и электро-атрибуты; технолог — габариты/U/
  размещение; и т.д. Дубли решаются как memory `rack-merge`
  (общий реестр by-design; авто-suggest «Связать» при совпадении tag
  как fallback для legacy).
- **Порты определяют межмодульную видимость:** объект несёт
  `ports[]` с типом (`power|data|fieldbus|pipe|duct|gas|...`).
  Объект всплывает в модуле, если у него есть порт нужного типа:
  data-порт → виден в СКС; полевая шина → в АСУТП; труба → в
  гидравлике; и т.д. Эмиттер — владелец объекта; потребитель —
  модуль дисциплины (контракт cross-module-events / project-storage,
  без сырого чужого LS).
- **Сборка ВСЕГО оборудования — только в технологе объекта**
  (tech-workspace / facility-inventory): единая точка агрегации BOM/
  инвентаря по всем дисциплинам.

**Контракт-файл (Ф-E задел, v0.60.729):** `shared/object-registry.js`
— чистый контракт + аксессоры (как Ф-A disciplines.js): typedef
`RegistryObject` (id/kind/tag/ownerModule/ports[]/disciplineAttrs),
`PORT_TYPES` (power|data|fieldbus|pipe|duct|gas), `objectPorts`,
`isVisibleToPort`/`objectsVisibleToPort` (порт-driven видимость),
`disciplineSlice` (read дисципл-среза, без мутации),
`duplicateTagGroups` (fallback dedup-suggest для memory:rack-merge).
БЕЗ доступа к LS, БЕЗ авто-записи, 0 потребителей (cache-safe).
Интеграция с project-storage швом + реконсиляция с tech-workspace/
rack-merge — следующие защищённые инкременты Ф-E.

**Ф-E интеграция #1 (v0.60.730):** `shared/object-registry-store.js`
— персистентность через project-storage шов, НОВЫЙ namespace
`raschet.project.<pid>.object-registry.objects.v1`. API
loadRegistry/saveRegistry/upsertObject/removeObject/
writeDisciplineSlice (merge disciplineAttrs без затирания чужих
срезов; объекта нет → no-op). Аддитивно (rack-config/tech-workspace/
scs-* не затронуты); copyProject/clearProjectData несут namespace
автоматически. 0 потребителей (cache-safe). Деривация реестра из
существующих rack/device-данных + wiring tech-workspace/rack-merge
— следующие инкременты Ф-E.

**Ф-E деривация (v0.60.731):** `deriveRegistry(pid)` в store —
read-only проекция RegistryObject[] из rack-config.instances.v1 +
порт-driven инференс (PDU/нагрузка→power→электрика; СКС-
устройства/связи→data→СКС) + дисципл-срезы (electrical/mechanical).
Чистое чтение: источники остаются владельцами (§5/§6), деривация =
координационный снимок, не мутирует/не пишет. 0 потребителей.
Дальше: wiring снимка в Конструктор/tech-workspace UI + опц.
материализация в namespace (явная, не авто) — следующие инкременты.

## 10. Пространственная база + жизненный цикл (memory D7/D8)

**Архитектор + конструктор → SPATIAL-BASE.** Появляются роли/
дисциплины, владеющие геометрией здания:
- `architectural` (архитектор) — НОВАЯ дисциплина: стены, помещения,
  площади, зоны, уровни, оси. Добавляется в `shared/disciplines.js`
  аддитивно, `ready:false` (как mechanical/data: 0 потребителей,
  cache-safe; движок — будущая фаза).
- structural (конструктор) — несущие/нагрузки ≈ существующая
  `mechanical`. Не дублировать сущность; при необходимости — переименуемая
  метка, не новый id (memory:user-params-sacred — id стабилен).
- Инженерные дисциплины НЕ создают стены/площади — только ССЫЛАЮТСЯ:
  размещение стоек/оборудования (tech-workspace, СКС «План зала»),
  no-cross-стены (memory:cable_routing), площади помещений/коридоров,
  clearances (memory:rack_clearances). Это BIM-federated паттерн:
  architectural model — носитель, MEP — наложение read-only подложкой
  (§3). Дубли геометрии решаются как §6 (общий per-project источник).

**Технолог-ГИП → владелец жизненного цикла.** Технолог проекта =
полное управление + ГИП на весь жизненный цикл объекта (инициация →
**сдача в эксплуатацию**). Это координатор над всеми дисциплинами
(§4), а не равный специалист: владеет lifecycle-статусом проекта,
сводным BOM/графиком/согласованиями (projects-view + tech-workspace —
его рабочее место, уточняет §6 «сборка всего оборудования —
у технолога»). Граница владения = акт ввода в эксплуатацию; после —
ответственность переходит модулю эксплуатации/O&M (отдельная фаза
ROADMAP, вне scope Конструктора).

Следствие для плана: §10 — задел контракта (дисциплина
`architectural` аддитивно в Ф-A-классе риска). Движок
пространственной базы и lifecycle-машина технолога — отдельные
будущие фазы (после Ф-G), высокий blast-radius, здесь не
реализуются.
