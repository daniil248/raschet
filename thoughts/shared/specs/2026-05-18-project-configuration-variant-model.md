# Спецификация: модель «Проект › Конфигурация › Вариант» (GE Tools)

## Executive Summary
В GE Tools «под-проект/мини-проект» (sketch с parentProjectId) ошибочно
открывается как полноценный **проект объекта** (полная карточка:
категория ЦОД, локация, реквизиты, ГИП-сводка, все вкладки). Это
нелогично: набор данных одной дисциплины — не объект-комплекс. Спека
вводит каноничную 3-уровневую модель и условный интерфейс карточки по
типу сущности. Без переноса данных (только тип + условный UI).

## Problem Statement
- Открытие «СКС» (sketch внутри `p_tyux2vnmz4`, 240 связей, 33 устр-в)
  показывает полную карточку объекта с ЦОД/Темиртау/Genesis — данные
  объекта, не относящиеся к одной дисциплине.
- Терминология размыта: «мини-проект», «под-проект», «конфигурация»
  использовались вперемешку; «проект» применялся и к комплексу, и к
  однодисциплинарному набору.
- Следствие — путаница пользователя и (ранее) рассогласование данных
  между модулями (последнее уже устранено `?project=`-швом v0.60.751).

## Канон терминологии (источник истины)
| Уровень | Термин | Что это | Интерфейс |
|---|---|---|---|
| 1 | **Проект** (объект/комплекс) | Мультидисциплинарный объект (ЦОД и т.п.). `kind:'object'` | ПОЛНАЯ карточка |
| 1' | **Проект** (1-дисциплинарный) | Самостоятельная одна дисциплина, без родителя, но полноценный проект. `kind:'discipline'`, `parentProjectId=null` | СЛИМ-карточка + свои мини-реквизиты |
| 2 | **Конфигурация** | Набор данных ОДНОЙ дисциплины ВНУТРИ проекта-объекта. `kind:'discipline'`, `parentProjectId=set` | СЛИМ-карточка, объект-свойства read-only от родителя |
| 3 | **Вариант** | Альтернатива внутри конфигурации (★ выбранный / 🔁 резервный), как подбор холода/ИБП. `variantRole` (8.0-A) | в списке «Варианты» слим-карточки |

Внутренние идентификаторы (`kind:'sketch'`, `s_`-id, namespace) НЕ
переименовываются — это терминология/тип уровня представления.

## Success Criteria
- Открытие конфигурации/1-дисц.проекта НИКОГДА не показывает
  объект-разделы (категория/локация/дерейтинг/реквизиты/ГИП-сводка/
  согласование/команда/сводная экономика-BOM по объекту).
- Везде в UI: «вариант» (3-й ур.), «конфигурация» (2-й/1' ур.),
  «проект» только для комплекса или явно 1-дисц. проекта.
- Существующие sketch открываются в слим-карточке без потери данных.
- Тип сущности виден и иммутабелен после создания.

## User Journey
1. **Создание.** `/projects/` «Новый» → выбор типа: «Объект/комплекс
   (мультидисциплина)» | «Одна дисциплина» (+выбор дисциплины). Из
   карточки объект-проекта «+ конфигурация дисциплины X» создаёт
   конфигурацию внутри него. Тип после создания не меняется.
2. **Открытие объекта** → полная карточка (как сейчас).
3. **Открытие конфигурации / 1-дисц. проекта** → СЛИМ-карточка:
   - Сводка + Оборудование этой дисциплины
   - Экономика этой дисциплины (не сводная по объекту)
   - Проверки этой дисциплины
   - Варианты (★/🔁) + Управление (переименовать/удалить/открыть в
     модуле; для конфигурации — «Открыть карточку родителя»)
   - Объект-свойства: конфигурация — read-only из родителя (штамп);
     1-дисц. проект — собственные мини-поля (шифр/заказчик/адрес).
4. **Переключение** вариантов/конфигураций — через слим-карточку и
   единый picker (8.0-B, секция «Варианты проекта»).

## Functional Requirements

### P0 — Must
- **FR1. Тип сущности.** Аддитивное поле на объекте проекта:
  `entityKind ∈ {'object','discipline'}` (+ существующий
  `parentProjectId`, `ownerModule`/дисциплина). Конфигурация =
  `entityKind:'discipline' && parentProjectId`. 1-дисц.проект =
  `entityKind:'discipline' && !parentProjectId`. Объект =
  `entityKind:'object'`. Backward-compat: отсутствие поля →
  выводить из (`kind:'sketch'`+parent ⇒ discipline/config;
  `kind:'full'` ⇒ object). Иммутабелен после создания.
- **FR2. Условный рендер карточки** (`apps/projects/project.js`):
  по `entityKind` показывать ПОЛНЫЙ или СЛИМ набор разделов.
  - Слим показывает ТОЛЬКО: Сводка+Оборудование дисциплины,
    Экономика дисциплины, Проверки дисциплины, Варианты+Управление.
  - Слим СКРЫВАЕТ: Категория объекта, Локация/дерейтинг, Реквизиты
    компании, Сводка ГИП, Согласование, Команда, сводная
    Экономика/BOM по объекту, «Модули и связи» (список всех
    дисциплин — уровень проекта).
- **FR3. Объект-свойства для слим.**
  - Конфигурация: объект/заказчик/адрес/локация/реквизиты —
    READ-ONLY, наследуются от `parentProjectId` (правило
    feedback_project_location — задаётся 1 раз на объекте; для
    штампа/отчётов). Не редактируются на слим-карточке.
  - 1-дисц. проект: собственные минимальные поля (шифр, заказчик,
    адрес для штампа). БЕЗ категории объекта/дерейтинга/ГИП.
- **FR4. Создание с выбором типа.** В `/projects/` «Новый» —
  явный выбор типа (Объект/комплекс | Одна дисциплина[+дисциплина]).
  «+ конфигурация» из карточки объект-проекта создаёт
  `entityKind:'discipline'`+parent. Тип immutable.
- **FR5. Терминология.** Везде в видимом UI: «вариант» /
  «конфигурация» / «проект» строго по канону (продолжение
  релейбла v0.60.752–753). Бейдж/чип показывает тип
  («📦 объект» / «🧩 конфигурация: <дисциплина>» / «вариант»).
- **FR6. Миграция существующих — ТОЛЬКО тип/UI.** Идемпотентный
  type-инференс при чтении (sketch+parent ⇒ конфигурация дисциплины
  `ownerModule`). LS-namespace и данные НЕ трогаются. Деструктивный
  перенос namespace (бывший 8.0-C) — **отклонён** (риск живым
  данным, ≈ноль выгоды: доступ уже унифицирован `?project=`-швом).

### P1 — Should
- **FR7.** Единый picker (8.0-B) маркирует тип и не показывает
  конфигурации/1-дисц. в общем списке «проектов-объектов» (плоско) —
  только в секции «Варианты проекта» / по родителю (согласовано с
  v0.60.349).
- **FR8.** Отчёты конфигурации берут штамп-реквизиты от родителя
  (или собственные у 1-дисц. проекта) — единый `composeReport`.

### P2 — Nice
- **FR9.** «Повысить» 1-дисц. проект до объекта (добавив дисциплины)
  и «привязать» самостоятельную конфигурацию к объект-проекту.
- **FR10.** Read-only блок «связи этой дисциплины с другими» на
  слим-карточке (если есть cross-discipline ссылки).

## Technical Architecture
- **Данные.** Аддитивное `entityKind` в метаданных проекта
  (`getools.projects.v1`), `typeof`-guard, preserve-on-miss; type
  инференс-фолбэк для legacy. Namespace данных без изменений.
- **Компоненты.** `apps/projects/project.js` (рендер карточки по
  типу — главный объём); `shared/project-storage.js` (хелперы
  `entityKindOf(p)`, создание с типом, иммутабельность);
  `apps/projects/projects.js` (создание/список — выбор типа,
  бейджи); `shared/app-header.js` (бейдж/чип типа — продолжение
  8.0-B). Вариант-роли — уже 8.0-A (`variantRole`).
- **Контракт.** Тип immutable; конфигурация наследует объект-props
  от родителя read-only (не дублирует, не рассогласует —
  feedback_project_location).
- **Cache-safe (§6a).** Новые экспорты project-storage → деплой
  экспортов отдельно от потребителей.
- **Verify.** Естественная загрузка `apps/projects/project.html`
  для object / configuration / 1-disc; consoleclean; данные
  существующего «СКС» (240 связей) читаются в слим-карточке.

## Non-Functional
- Клиентский LS-app; без бэкенда. Производительность не меняется.
- Нулевой риск данным (FR6: только тип/UI; без миграции namespace).
- Аддитивно/идемпотентно; user-params-sacred.

## Out of Scope
- Деструктивное слияние namespace конфигураций внутрь проекта
  (бывший ROADMAP 8.0-C) — отклонено.
- Бэкенд/мульти-пользовательская модель проектов (Phase 41).
- «Повышение/привязка» (FR9) — P2, отдельная под-фаза.

## Open Questions (для реализации)
- Точный список разделов «Управление» слим-карточки (удаление
  конфигурации = soft-delete её scoped-ключей; не трогать родителя).
- Нужен ли 1-дисц. проекту блок «Локация» для climate-дисциплин
  (cooling/meteo)? Сейчас вне слим; решить при появлении 1-дисц.
  climate-проекта (вероятно — мини-поле координат только тогда).
- Иконографика бейджей типа (📦/🧩/★) — финал на этапе UI.
