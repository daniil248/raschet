Концепция

В реальном проекте часто несколько одинаковых стоек: SR01, SR02, ..., SR08. Создавать 8 отдельных карточек на схеме — захламляет canvas. Решение: группа потребителей-контейнер, где одна карточка представляет N экземпляров.

Модель «Shell»

Один из реальных потребителей (с самым младшим обозначением — например SR01) становится оболочкой (shell). Остальные (SR02..SR08) числятся «внутри». На схеме видна только оболочка — карточка с маркером ×N (стопка карточек с peek).

Свойства наследуются

По модели пользователя: «свойства расположенных внутри объектов связаны с основным (кроме обозначений)».

  • Tag/name — индивидуальные для каждого экземпляра
  • demandKw, cosφ, фаза, напряжение, subtype — наследуются от shell
  • В инспекторе экземпляра электрические поля показаны read-only с маркером «🔗 Электрические параметры наследуются от оболочки»

Изменения в shell автоматически распространяются на все экземпляры — через getter-helpers (effectiveDemandKw, effectiveCosPhi и т.д. в shared/zones.js).

Как создать группу-контейнер

Шаг 1. Размещение первого элемента

Drag неразмещённого потребителя (стойка SR01) с панели «Неразмещённые» на холст → встаёт как обычная карточка.

Шаг 2. Drop ещё одного похожего → авто-merge

Перетащите второй потребитель (SR02) поверх первого. Если параметры совместимы — auto-merge:

  • Совместимы = одинаковые: consumerSubtype, phase, voltageLevelIdx, cosPhi (±5%), demandKw (±5%). Не individual-mode.
  • Карточка на холсте превращается в контейнер ×2 (стопка с peek-эффектом сзади).
  • Tag и name контейнера = у первого по сортировке элемента (SR01).
  • Внутри: оба элемента (SR01 + SR02). Каждый можно открыть отдельно.

Несовместимые остаются отдельными карточками + toast-подсказка про ручную связь.

Шаг 3. Третий, четвёртый и т.д.

Drop SR03, SR04, ... поверх контейнера → каждый добавляется в группу. Контейнер становится ×3, ×4, ...

Через Группа-tab (ручная связь)

Если параметры разные, но физически это один объект (например технолог дал стойке другие электрические параметры), используйте ручную связь:

  1. Откройте свойства одного потребителя
  2. Вкладка 👥 Группа → секция «🔗 Связать с существующими»
  3. Перетащите кандидата в нужный слот — связь устанавливается без проверки compatibility

Cross-discipline reconciliation: «выбрать правду»

Сценарий: электрик уже разместил группу стоек SR1-SR8 (по своей логике), технолог добавил похожие стойки SR01-SR08 (с другими обозначениями). Это один и тот же физический объект, но представлен дважды.

Решение: через ручную связь в Группа-tab каждого потребителя. После связывания source-узел исчезает с canvas — остаётся ТОЛЬКО shell как «правда» (canonical truth). Source стоит в state.nodes но скрыт, properties наследуются.

Полный merge с удалением дубликата (v0.59.813)

В Группа-tab инспектора группы появилась секция «🔀 Похожие группы в проекте» — auto-detect других consumer-групп с тем же subtype и близким count (±20%).

Click 🔀 Объединить на нужной группе → диалог:

  • ОСТАВИТЬ: текущая группа (та, в которой вы сейчас) — становится «правдой»
  • УДАЛИТЬ: выбранная другая — её слоты перенесутся в текущую
  • Tag-match: alias из другой группы с тем же tag, что в текущей — не дублируется (пропускается)
  • Уникальные tag — добавляются как новые слоты, count++

Действие необратимо без Ctrl+Z. После merge'а удалённая group-нода исчезает, все её back-references очищаются.

Это и есть то «выбрать какой элемент является правдой» — пользователь сам решает кто остаётся canonical, дубликат удаляется.

Через Группа-tab инспектора

Откройте свойства любого consumer-узла (групповой потребитель) → вкладка 👥 Группа:

  • Список «слотов» — позиций внутри shell. Каждый слот может быть привязан к конкретному экземпляру или быть anonymous.
  • Picker «🔗 Связать с существующими» — список одиночных consumer'ов, которые можно перетащить в свободный слот.
  • Кнопки на linked-слоте: ↗ Извлечь (split-out — слот удаляется, count--), ✂ Разъединить (slot становится anonymous).

Click-навигация

В Группа-tab каждая строка слота имеет:

  • Click по строке (вне кнопок) → открыть свойства привязанного узла
  • Click по 🔗 (зелёный кружок) → центрировать камеру на shell на схеме (если узел на другой странице — переключение)
  • Кнопки ↗ (split-out) / ✂ (unlink)

Также: Ctrl+click по объекту в Реестре → центрирование на схеме (см. навигацию в Конструкторе).

Auto-tag = младшее обозначение

Эффективное обозначение группы (отображаемое на канвасе и в модалке) = обозначение первого экземпляра в естественной сортировке (SR01 < SR02 < SR10 через localeCompare numeric:true). При добавлении SR00 — оно автоматически становится новым «лицом» группы.

⚠ Raw tag группы (внутреннее n.tag) не меняется — остаётся уникальным для save/load. Меняется только display-derived через effectiveTag().
